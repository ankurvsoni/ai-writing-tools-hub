# 2026-03-08

## Session learnings (affiliate-content-hub)

- Confirm project target first by checking:
  - active git branch
  - Vercel linked project (`.vercel/project.json` / `vercel link`)
  - custom-domain alias map (`vercel alias ls`)
- A successful deploy to the wrong project/domain is still a failure for the user.
- `reviews.orkolabs.com` can lag behind if still aliased to an older deployment; fix with:
  - `vercel alias set <latest-deployment> reviews.orkolabs.com`
- Styling bug root cause:
  - page wrapper used `.content`, but Amazon link CSS targeted `.container`.
  - Fix by targeting both `.content` and `.container`.
- Duplicate cart icons came from two sources:
  - inline icon in CTA text + CSS `::before` icon.
  - Keep one source only.
- Duplicate header/disclosure UX issues came from template + markdown overlap.
  - Avoid rendering repeated H1/disclosure blocks in page template when content already includes them.
- For affiliate pages, generic lines are immediately visible to users; prefer concrete buyer-language:
  - what it is
  - best for (specific persona)
  - what stands out (specific differentiator)
  - before-you-buy checks
  - skip-if criteria
- Operational guardrail: after any fix, verify both:
  - direct Vercel deployment URL
  - production/custom domain URL

## Process improvements to keep

- Before claiming "deployed", run this checklist:
  1. `git rev-parse --abbrev-ref HEAD`
  2. `vercel whoami`
  3. `vercel alias ls --scope ankurvsonis-projects | grep <domain>`
  4. open/fetch both deployment URL and live domain URL for the same page
- When user reports mismatch with screenshot, assume domain alias mismatch first, then cache.

## Additional durable learnings (late session)

- User preference: avoid crowded card headers; do not show volatile star/rating counts inline in top-pick headers.
- Better pattern for trust signals:
  - keep header minimal (`Product — short descriptor — Check on Amazon`)
  - move reliability to body as `Review confidence: High/Medium/Low`.
- Implemented visual confidence pills in product cards:
  - High (green), Medium (amber), Low (rose)
  - Parser maps `Review confidence:` lines to semantic classes in rendered HTML.
- Content quality lesson:
  - over-defensive “evidence-only” wording can become robotic and reduce readability.
  - preferred approach is hybrid: human-readable summaries + conservative claims.
- Operations lesson:
  - avoid parallel navigation calls on same browser tab during visual QA; navigate and snapshot sequentially to prevent accidental last-page-only checks.
- Mobile UX observation:
  - current nav works but is crowded on 375px widths; candidate future polish is collapsing nav links behind a compact menu.

## Additional learnings (late-night iteration)

- Bulk rewrite risk: automated “confidence-first re-rank” can silently reintroduce generic boilerplate across multiple pages; always sample-check at least 3 pages from different categories before deploy.
- Content quality rule reinforced: product cards must be category-specific and item-specific; generic fallback lines should be treated as a defect.
- For fashion pages (dresses/workout sets), buyers need concrete fit-oriented checks (opacity, length, waist/torso fit, fabric thickness) rather than abstract confidence language.
- Confidence badges are useful, but require a clear legend and clean subtitle copy; avoid awkward “listing” phrasing in card headers.
- Header confidence approach accepted by user:
  - emoji marker in card title (🟢/🟡/🔴)
  - concise legend above top picks.
- Visual polish lesson: confidence pill/dot alignment can look off on-device; tune line-height/vertical alignment using real screenshot feedback, not assumptions.
- SEO/content lesson: include recognizable product/brand names in card titles when reliable and concise.
- Decision/process lesson: when user asks “did you research or just lie?”, explicitly disclose method limitations and switch to evidence-grounded copy immediately.
- Operational lesson: alias commands using “latest deployment” can accidentally target a still-building deployment; prefer pinning a known-ready deployment URL for custom domain mapping.
- QA lesson reinforced: duplicate-line checks are not enough; cross-category content contamination can still pass. Added a category-consistency QA gate (`scripts/check-card-consistency.py`, `npm run qa:consistency`) to catch mismatched intent phrases (e.g., standing-desk language inside cable-management pages).
